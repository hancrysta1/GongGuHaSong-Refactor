# k6 부하 테스트

> 재고 동시성 + SAGA 보상 트랜잭션을 Before/After 비교로 정량 검증

<br>

## 1. 왜 부하 테스트를 하는가

금액, 잔액 같은 금전 데이터는 한 건이라도 꼬이면 안 된다. 로직 단위로는 정합성을 보장하더라도, 실제 부하 환경에서 여러 요청이 동시에 몰렸을 때도 안전한지는 돌려봐야 알 수 있다. 특히 포인트 적립과 차감이 동시에 같은 잔액을 건드리는 상황은 평소에는 잘 안 생기지만, 생겼을 때 잔액이 꼬이면 치명적이기 때문에 일부러 극한 부하를 걸어서 확인했다.

확인하고 싶었던 건 세 가지였다.
- 재고가 마이너스로 빠지거나 초과 판매되지 않는가
- 결제 실패 시 포인트가 유실되지 않는가
- 마이너스 잔액이 발생하지 않는가

실제로 SAGA 보상이 단건에서는 잘 되지만, 300명이 동시에 보상 요청을 보내면 커넥션 풀이 고갈되어 보상 자체가 실패하는 걸 부하 테스트에서 발견했다. 이런 건 부하를 걸어보지 않으면 모른다.

정합성이 검증되면, 트래픽이 늘어날 때는 인프라(커넥션 풀 사이즈, 인스턴스 수)만 늘리면 된다.

<br>

## 2. 테스트 설계

두 가지 문제를 각각 Before/After로 검증한다.

| | Before | After |
|---|---|---|
| **재고 동시성** | 재고 100개에 300~3000명 동시 주문 → 초과 판매? | 원자적 차감 → 재고만큼만 성공? |
| **SAGA 보상** | 장애 주입 10% + 보상 없음 → 포인트 유실? | 장애 주입 10% + 보상 있음 → 포인트 복구? |

### 왜 부하 + 장애 주입을 같이 하는가

SAGA 보상의 기능 검증만 필요하다면 장애 주입만으로 충분하다. 하지만 실제 운영에서 장애는 트래픽이 몰릴 때 발생한다. 카드사 API가 평소에는 정상이지만 블랙프라이데이에 타임아웃이 터지고, DB 커넥션 풀이 포화되면서 결제 중간에 실패하는 식이다.

그래서 두 조건을 결합한다:
- **장애 주입 10%**: 코드 레벨에서 결제 도중 강제 예외 발생 → SAGA 보상 대상을 만든다
- **부하 (50→100→200→300명)**: 동시 요청이 증가하는 환경에서 보상 로직이 안정적으로 동작하는지 확인한다

장애 주입 없이 부하만 걸면 로컬 Docker 환경에서는 서비스 간 통신이 거의 실패하지 않아 SAGA 보상이 개입할 일이 적다. 반대로 부하 없이 장애 주입만 하면 "1명이 결제할 때는 되는데 300명이 동시에 보상 요청을 보내면?" 을 알 수 없다.

### 1인 1결제 원칙

처음 SAGA 테스트를 설계할 때는 300명의 VU가 1,000명의 유저를 돌려쓰며 반복 결제하는 구조였다. 이 설계로 55,389건을 돌렸을 때 `point_inconsistency`가 1건 발생했다.

원인을 추적해보니 같은 유저가 0.02초 간격으로 반복 결제하면서, 이전 결제의 잔액 조회와 다음 결제의 잔액 조회 사이에 다른 트랜잭션이 끼어드는 거짓 양성이었다. 현실에서 한 사람이 0.02초마다 결제를 반복하는 건 있을 수 없다.

그래서 SAGA 테스트는 **1인 1결제**로 재설계했다. VU 번호와 iteration을 조합해 매번 새로운 유저를 사용한다.

```javascript
// Before: 같은 유저 반복 → 거짓 양성
const userId = `saga-user-${(__VU - 1) % 1000 + 1}`;

// After: 1인 1결제 → 유저 완전 분리
const userId = `saga-${__VU}-${__ITER}`;
```

```bash
k6 run load-test/4-saga-after.js
```

## 3. 재고 동시성 — Before vs After

### Before: Read-Check-Write (잠금 없음)

재고 100개 상품에 300명이 동시 주문. `findById` → 재고 체크 → `save`가 분리되어 있어 잠금 없음.

```
k6 run 1-stock-before.js

  성공: 300건 / 300건 (100건만 성공해야 정상)
  최종 재고: 33개 (0이어야 정상)
```

**300명 전원 성공, 재고 33개 남음.** 재고 100개짜리 상품을 300명에게 다 팔았다. 300명이 동시에 `stock=100`을 읽고 전부 재고 체크를 통과했기 때문이다.

```
VU-1: stock=100 읽음 → 100-1=99 저장
VU-2: stock=100 읽음 → 100-1=99 저장  ← 같은 100에서 뺌
VU-3: stock=100 읽음 → 100-1=99 저장  ← 또 같은 100에서 뺌
→ 3건 차감됐는데 재고는 99
```

### After: 원자적 재고 차감

MongoDB `findAndModify` 또는 조회+차감을 원자적으로 처리. 재고 부족 시 즉시 거절.

```
k6 run 2-stock-after.js

  성공: 100건 / 300건
  최종 재고: 0개
```

**정확히 100명만 성공, 재고 정확히 0.** 200명은 "재고 부족"으로 거절.

### 비교

| | Before | After |
|---|---|---|
| 재고 100개에 300명 주문 | 300명 전원 성공 (초과 판매) | 100명만 성공 |
| 최종 재고 | 33개 (67개만 차감) | 0개 (정확히 100개 차감) |
| 원인 | 조회-체크-차감 분리 → race condition | 원자적 차감 → 순차 처리 |

<br>

## 4. SAGA 보상 트랜잭션 — Before vs After

### 테스트 조건

- 50 → 100 → 200 → 300명 단계별 (각 1분, 총 약 6분)
- 1인 1결제
- **장애 주입 10%**: 서버 내부에서 포인트 차감 후 DB 저장 전에 10% 확률로 예외 발생
- 검증: 결제 실패 시 포인트가 복구되는가?

장애 주입 코드:
```java
// PaymentService.createPayment() — STEP 2와 STEP 3 사이
if (Math.random() < 0.1) {
    throw new RuntimeException("장애 주입: 결제 저장 전 서버 오류");
}
// → 포인트는 이미 차감된 상태. SAGA 없으면 유실.
```

### ① Before: 장애 주입 10% + 보상 없음

```
k6 run 3-saga-before.js

  총 결제:     7,263건
  성공률:      71.78%
  결제 실패:   ~726건 (장애 주입 10%)
  포인트 유실:   656건 (유실률 90.4%)
  마이너스 잔액:   0건
```

10% 장애 주입으로 ~726건이 "포인트 차감 후 결제 저장 전"에 실패했고, 보상 로직이 없으므로 **656건(90.4%)의 포인트가 차감된 채 복구되지 않았다.**

### ② SAGA only: 장애 주입 10% + SAGA 보상 (Outbox 없음)

SAGA 보상을 적용하면 개선될 거라 기대했다. 하지만 결과는 예상과 달랐다.

```
k6 run 4-saga-after.js

  총 결제:     7,128건
  성공률:      63.53%
  결제 실패:   ~712건 (장애 주입 10%)
  포인트 유실:   691건 (유실률 97.1%)
  마이너스 잔액:   0건
```

**오히려 악화됐다.** Before보다 유실률이 90.4% → 97.1%로 올라갔고, 성공률도 71.78% → 63.53%로 떨어졌다.

서버 로그를 확인하니 원인이 명확했다:

```
HikariPool-1 - Connection is not available, request timed out after 30001ms
```

**DB 커넥션 풀 고갈.** SAGA 보상이 실행될 때마다 point-service에 추가 REST 호출이 발생한다. 300명이 동시에 결제하면서 동시에 보상까지 발생하면, point-service로의 요청이 2배로 늘어나 MySQL 커넥션 풀이 포화된다.

```
Before:  결제 실패 → 끝 (추가 부하 없음)
SAGA:    결제 실패 → 보상 호출 → point-service 부하 증가 → 커넥션 풀 고갈
         → 보상도 실패, 정상 결제도 영향 → 성공률까지 하락
```

SAGA 보상 자체가 시스템에 부하를 추가하는 구조이기 때문에, **보상 호출이 실패했을 때 재시도할 메커니즘이 없으면 오히려 상황이 악화**된다.

이 경험에서 CompensationOutbox의 필요성을 느꼈다. 보상을 즉시 호출하지 않고, 실패 건을 로컬 DB에 저장한 후 트래픽이 줄었을 때 재시도하는 방식이다.

### ③ SAGA + Outbox: 장애 주입 10% + SAGA 보상 + CompensationOutbox 재시도

SAGA 보상이 실패하면 Outbox 테이블에 저장, 30초 폴링으로 최대 5회 재시도.

```
k6 run 4-saga-after.js

  총 결제:     7,408건
  성공률:      73.74%
  실시간 누락:   854건 (rollback_missing)
  마이너스 잔액:   0건

  ===k6 test 후 DB 조회 최종 - 유실 0건
```

실시간 측정(`rollback_missing`)에서는 854건이 유실로 감지되었다. 이는 SAGA 보상 호출이 point-service에 부하를 추가하면서, 보상 처리가 k6의 잔액 조회 시점보다 늦게 완료되기 때문이다.

그러나 테스트 종료 후 60초 대기(Outbox 30초 폴링 × 2회) 후 전체 유저의 DB 잔액을 재조회한 결과, **최종 유실 0건**. SAGA 보상이 즉시 실패하더라도 Outbox가 트래픽이 줄어든 후 재시도하여 전부 복구한 것이다.

### 비교

| | ① Before (보상 없음) | ② SAGA only (Outbox 없음) | ③ SAGA + Outbox |
|---|---|---|---|
| 총 결제 | 7,263건 | 7,128건 | 7,408건 |
| 성공률 | 71.78% | 63.53% | 73.74% |
| 장애 주입 실패 | ~726건 | ~712건 | ~740건 |
| 실시간 유실 (`rollback_missing`) | 656건 (90.4%) | 691건 (97.1%) | 854건 |
| **DB 최종 유실** | **656건** | **691건** | **0건** |
| 마이너스 잔액 | 0건 | 0건 | 0건 |

### 결과 해석

**① → ②에서 유실이 늘어난 이유**

k6 테스트 시나리오는 0.1초 간격으로 결제 요청을 반복하면서 300명이 동시에 부하를 건다. 이 환경에서 SAGA 보상이 실행되면 결제 실패 건마다 point-service에 복구 REST 호출이 추가로 발생하고, 이미 포화 상태인 DB 커넥션 풀에 부하가 더해져 보상 호출 자체가 타임아웃된다. 즉 SAGA가 문제가 아니라, **테스트 환경의 커넥션 풀 한계**에서 비롯된 현상이다.

실제 운영에서는 HikariCP 커넥션 풀 사이즈를 늘리거나, 서비스 인스턴스를 스케일아웃하거나, 보상 호출을 비동기로 전환하는 등의 방법으로 이 문제를 완화할 수 있다. 이 프로젝트에서는 로컬 Docker 단일 인스턴스 환경의 제약으로 커넥션 풀 튜닝까지는 진행하지 않았다.

**그럼에도 Outbox가 필요한 이유**

커넥션 풀을 아무리 튜닝해도, 인프라 장애(DB 다운, 네트워크 단절, 서비스 크래시)는 언제든 발생할 수 있다. Outbox는 이런 **최악의 인프라 상황에서도 보상이 유실되지 않도록** 보장하는 마지막 안전망이다. 보상 호출이 실패하면 로컬 DB에 저장해두고, 트래픽이 줄거나 장애가 복구된 후 30초 폴링으로 재시도한다.

③의 결과에서 이를 확인할 수 있었다. 실시간 측정에서는 854건이 유실로 감지되었지만, Outbox 재시도 후 최종 유실은 0건이었다. 즉시 완벽하지 않아도 최종적으로 정합성이 수렴하는 **최종 일관성(Eventual Consistency)** 을 직접 경험한 케이스였다.

**Outbox에서도 실패하면?**

Outbox는 최대 5회 재시도하고, 그래도 실패하면 `FAILED`로 남긴다. 이 경우 Kafka Dead Letter Queue(DLQ)에 적재하여 별도 컨슈머가 처리하거나, 일 1회 정합성 배치로 payment 기록과 point 이력을 대조하여 불일치 건을 자동 보정하는 방식으로 보완할 수 있다. 최종적으로 자동 복구가 불가능한 건은 운영팀에 알림을 보내 수동으로 처리한다.

| 계층 | 역할 | 실패 시 |
|------|------|---------|
| SAGA 보상 | 즉시 복구 시도 | Outbox에 저장 |
| CompensationOutbox | 30초 폴링 재시도 (최대 5회) | FAILED 전환 |
| DLQ / 배치 | 별도 컨슈머 또는 일 1회 대조 | 운영팀 알림 |

## 5. 포인트 동시성 — MySQL SELECT FOR UPDATE

### 왜 이 테스트가 필요한가

주문이 발생하면 결제 API에서 포인트 차감(`usePoints`, 동기)이 실행되고, 거의 동시에 Kafka를 통해 포인트 적립(`earnPoints`, 비동기)이 point-service에 도착한다. 두 요청이 같은 유저의 같은 포인트 행을 동시에 수정하는 상황이 생긴다.

Kafka는 비동기라 적립이 늦게 오면 괜찮지 않나 싶을 수 있는데, Kafka 소비 지연은 수 밀리초 수준이라 결제 차감과 거의 동시에 도착한다. 그리고 순서가 바뀌어도(적립이 먼저든 차감이 먼저든) 결과는 같아야 한다.

이걸 가능하게 하는 게 `SELECT FOR UPDATE`(비관적 락)다. 같은 유저의 포인트 행에 대해 먼저 도착한 요청이 트랜잭션을 끝낼 때까지 다음 요청을 대기시킨다.

```
결제 API:  usePoints(-100P)  → SELECT FOR UPDATE → 행 잠금 → 차감 → COMMIT → 잠금 해제
Kafka:     earnPoints(+100P) → SELECT FOR UPDATE → 대기...                → 잠금 획득 → 적립 → COMMIT
```

카프카에서 대기하는 게 아니라, point-service의 DB에서 대기하는 거다. 카프카는 메시지를 전달하는 역할이고, 정합성은 DB 락이 책임진다. 차감이 먼저 실행돼서 잔액이 0이 되어도 적립은 대기하다가 차감이 끝난 후에 실행되니까 최종 잔액은 맞고, 마이너스도 나지 않는다.

처음에는 적립(`earnPoints`)에 낙관적 락(`@Version`)을 적용했다가 차감(`usePoints`)의 비관적 락과 충돌하여 적립이 실패하는 문제가 있었다. 결국 둘 다 비관적 락으로 통일했고, 이 과정은 [PAYMENT_TROUBLESHOOTING.md 7장](PAYMENT_TROUBLESHOOTING.md)에 정리했다.

### 테스트 조건

- 50명 유저에 300 VU가 5분간 동시 요청
- 절반은 100P 적립(earnPoints), 절반은 100P 차감(usePoints)
- 같은 유저의 잔액을 적립과 차감이 동시에 수정하는 상황
- 검증: 마이너스 잔액이 발생하는가

평소에는 적립과 차감이 동시에 같은 잔액을 건드리는 일이 드물지만, 금액 데이터는 한 건이라도 꼬이면 안 되기 때문에 일부러 극한 부하를 걸어서 확인했다.

### 결과

```
k6 run 5-point-concurrency.js

  300 VU × 5분
  총 요청:       213,185건
  마이너스 잔액:   0건
  최종 마이너스:   0명 (teardown 전수 조사)
```

300명이 5분간 21만 건을 동시에 적립/차감해도 마이너스 잔액 0건. `SELECT FOR UPDATE`가 동시 수정 환경에서도 잔액 정합성을 보장하는 것을 확인했다. 성공률이 50%인 건 차감 요청 중 잔액 부족으로 정상 거절된 건이 포함되어 있기 때문이다.

## 6. DB 비관적 락 vs Redis 분산 락 — 스케일아웃 환경 비교

### 왜 이 테스트가 필요한가

5장에서 `SELECT FOR UPDATE`(비관적 락)가 단일 인스턴스에서 동시성을 보장하는 것을 확인했다. 그런데 서비스 인스턴스가 늘어나면 어떻게 될까?

DB 비관적 락의 동작 방식을 다시 보자:

```
[인스턴스 1] usePoints() → HikariPool에서 커넥션 획득 → SELECT ... FOR UPDATE → 행 잠금 → 대기
[인스턴스 2] usePoints() → HikariPool에서 커넥션 획득 → SELECT ... FOR UPDATE → 행 잠금 대기 (커넥션 점유 중)
[인스턴스 3] usePoints() → HikariPool에서 커넥션 획득 → SELECT ... FOR UPDATE → 행 잠금 대기 (커넥션 점유 중)
```

단일 인스턴스에서는 하나의 커넥션 풀(기본 10개)에서 순차적으로 락을 잡고 풀지만, 인스턴스가 3대면 커넥션 풀도 3개(총 30개)가 된다. 같은 유저의 같은 행에 30개 커넥션이 동시에 `FOR UPDATE`를 시도하면, 락을 획득한 1개를 제외한 29개가 **DB 커넥션을 점유한 채 대기**한다. 트래픽이 더 몰리면 커넥션 풀이 고갈되고, 락과 무관한 다른 쿼리(조회, 적립)까지 영향을 받는다.

Redis 분산 락은 이 문제를 구조적으로 해결한다:

```
[인스턴스 1] usePoints() → Redis LOCK 획득 → DB 커넥션 획득 → 작업 → COMMIT → Redis UNLOCK
[인스턴스 2] usePoints() → Redis LOCK 대기 (DB 커넥션 점유 없음)
[인스턴스 3] usePoints() → Redis LOCK 대기 (DB 커넥션 점유 없음)
```

락 대기가 Redis에서 일어나기 때문에 DB 커넥션을 점유하지 않는다. 인스턴스가 몇 대든 Redis가 중앙에서 순서를 관리하고, DB 커넥션은 실제 작업할 때만 잠깐 사용한다.

이 차이가 실제로 성능에 얼마나 영향을 미치는지, 단일 인스턴스와 멀티 인스턴스에서 각각 부하를 걸어 비교했다.

### 테스트 조건

- 30명 유저 × 2그룹 (DB 락 전용 / Redis 락 전용), 초기 포인트 100만P
- Phase 1: DB 비관적 락 (2분, 50→300 VU ramp-up)
- Phase 2: 쿨다운 (30초)
- Phase 3: Redis 분산 락 (2분, 50→300 VU ramp-up)
- 검증: 처리량(TPS), 지연시간(p95), 성공률, 데이터 정합성(마이너스 잔액)

### 단일 인스턴스 결과

point-service 1대, 직접 호출 (포트 8084)

```
k6 run 6-db-vs-redis-lock.js    # 4분 30초

  [DB 비관적 락]   처리량: 37,487건 (312 TPS)  avg: 217ms  p95: 747ms   성공률: 100%  마이너스 잔액: 0명
  [Redis 분산 락]  처리량: 22,293건 (186 TPS)  avg: 584ms  p95: 1.57s   성공률: 100%  마이너스 잔액: 0명
```

단일 인스턴스에서는 DB 비관적 락이 압도적이다. 커넥션 풀 하나에서 `SELECT FOR UPDATE`가 바로 처리되는 반면, Redis 락은 락 획득/해제마다 Redis 네트워크 왕복(RTT)이 추가된다. 처리량 68% 높고, p95 지연 53% 낮다.

### 멀티 인스턴스 3대 결과

point-service 3대 + nginx 로드밸런서 (포트 8090), 동일 MySQL/Redis 공유

```
k6 run 7-multiinstance-lock.js    # 4분 37초

  [DB 비관적 락]   처리량: 30,145건 (251 TPS)  avg: 278ms  p95: 1.09s   성공률: 100%  마이너스 잔액: 0명
  [Redis 분산 락]  처리량: 40,580건 (338 TPS)  avg: 292ms  p95: 704ms   성공률: 100%  마이너스 잔액: 0명
```

**3대로 스케일아웃하자 성능이 역전됐다.**

### 비교

| 지표 | 단일 (DB / Redis) | 3대 (DB / Redis) | DB 변화 | Redis 변화 |
|------|-------------------|-------------------|---------|------------|
| TPS | 312 / 186 | 251 / 338 | -19.6% | **+81.7%** |
| 처리량 | 37,487 / 22,293 | 30,145 / 40,580 | -19.6% | **+82.0%** |
| avg 지연 | 217ms / 584ms | 278ms / 292ms | +28.1% | **-50.0%** |
| p95 지연 | 747ms / 1.57s | 1.09s / 704ms | +45.9% | **-55.2%** |
| 성공률 | 100% / 100% | 100% / 100% | 동일 | 동일 |
| 마이너스 잔액 | 0명 / 0명 | 0명 / 0명 | 동일 | 동일 |

### 결과 해석

**DB 비관적 락이 스케일아웃에서 성능이 떨어지는 이유**

인스턴스가 3대로 늘어나면 HikariCP 커넥션 풀도 3배가 된다. 같은 유저에 대한 요청이 3개 인스턴스에 분산되면, 각 인스턴스의 커넥션이 동시에 같은 행에 `FOR UPDATE`를 시도한다. 락을 획득하지 못한 커넥션은 DB 레벨에서 대기하면서 커넥션을 계속 점유한다. 이 대기 커넥션이 쌓이면 풀이 고갈되고, 락과 무관한 조회 요청까지 지연된다.

**Redis 분산 락이 스케일아웃에서 오히려 좋아지는 이유**

Redis 락은 DB 커넥션과 분리되어 있다. 락 대기는 Redis에서 일어나고, DB 커넥션은 락을 획득한 후에만 사용한다. 인스턴스가 늘어나도 DB 커넥션 점유 시간이 짧게 유지되기 때문에, 총 처리 용량이 인스턴스 수에 비례하여 늘어난다. 단일에서 Redis RTT가 병목이었지만, 3대에서는 그 오버헤드보다 DB 커넥션 효율이 주는 이득이 훨씬 크다.

**정합성은 양쪽 모두 완벽**

두 방식 모두 마이너스 잔액 0건으로, 동시성 제어 자체는 정확하다. 차이는 처리량과 지연, 즉 성능에서만 나타난다.

### 결론

단일 인스턴스 환경에서는 DB `SELECT FOR UPDATE`가 단순하고 빠르다. 하지만 서비스를 스케일아웃하면 커넥션 풀 경합이 병목이 되어 성능이 저하된다. Redis 분산 락은 락 대기를 DB 밖으로 빼내어 커넥션 풀을 보호하므로, 인스턴스가 늘어날수록 오히려 처리 성능이 향상된다.

이 프로젝트에서는 DB 비관적 락을 기본으로 사용하되, Redis 분산 락을 함께 구현하여 스케일아웃 시 전환할 수 있도록 설계했다. 두 방식의 트레이드오프를 정량적으로 검증한 결과를 바탕으로, 운영 환경의 인스턴스 수에 따라 적합한 전략을 선택할 수 있다.
