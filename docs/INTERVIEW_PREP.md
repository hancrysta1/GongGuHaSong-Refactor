# 예상 질문 & 답변 가이드

---

## 아키텍처

Q. 왜 MSA로 했나?
> 원래 모놀리식이었다. 리팩토링하면서 결제는 MySQL, 상품은 MongoDB, 검색은 Elasticsearch를 써야 했는데, 모놀리식이면 하나의 DB에 묶여서 이런 선택이 불가능하다. 서비스별 독립 배포와 기술 선택을 위해 MSA로 전환했다.

Q. point와 payment를 왜 나눴나? 통일하면 안 되나?
> 통일할 수 있다. 하지만 나눈 이유가 있다. 포인트 적립은 "나중에 줘도 되는" 비동기(Kafka), 차감은 "잔액 부족이면 즉시 중단" 동기(REST)이다.

Q. (꼬리) 적립도 REST로 통일하면 안 되나?
> 되긴 된다. 하지만 주문 확정 시 order-service가 point(적립) + payment(재고 예약) + search(랭킹)을 전부 REST로 호출하면, 3개 서비스 응답을 다 기다려야 한다. point-service가 느리면 주문 전체가 느려지고, search-service가 죽으면 주문 자체가 실패한다. Kafka(비동기)로 이벤트를 발행하면 order-service는 즉시 응답하고, 나머지는 각자 알아서 처리한다. 적립이 1초 늦어도 사용자는 모르고, search-service가 죽어도 주문에 영향이 없다. 반면 차감은 "잔액이 충분한가"를 즉시 확인해야 결제를 진행할지 중단할지 결정할 수 있으므로 동기여야 한다.

Q. 이거 혼자 다 했나?
> 원본은 4명이 만든 교내 프로젝트이다. 리팩토링은 1인으로 진행했고, 코드 작성과 설정은 AI 도구를 활용했다. 내가 직접 한 건 기술 선택 판단과 결과 검증이다. 예를 들어 MongoDB에서 동시성 문제가 생겼을 때, MongoDB가 원래 금전 도메인에 맞지 않는 DB라는 걸 특성에서 파악하고 MySQL로 전환하기로 결정한 건 내 판단이었다.

## SAGA & 동시성

Q. SAGA가 뭔가? 왜 필요했나?
> 이전 프로젝트(급여 정산 서비스)에서는 같은 DB 안에서 `@Transactional`의 트랜잭션 전파(Propagation.REQUIRED)로 여러 클래스의 연산을 하나의 트랜잭션으로 묶었다. 하나라도 실패하면 전부 롤백되니까 문제가 없었다. 그런데 MSA에서는 이게 안 통한다. `@Transactional`은 하나의 DB 커넥션 안에서만 동작하는 로컬 트랜잭션이기 때문이다. 결제 시 포인트 차감(point-service MySQL)과 결제 기록 저장(payment-service MySQL)이 서로 다른 DB라서, 트랜잭션 전파가 불가능하다. 포인트는 차감됐는데 결제 저장에서 실패하면 포인트만 날아간다. 이걸 직접 겪고 "서비스 경계를 넘는 트랜잭션은 직접 보상해야 한다"는 걸 체감해서 SAGA를 도입했다. SAGA는 각 단계를 독립적으로 커밋하되, 실패 시 이전 단계를 역순으로 보상하는 패턴이다.

Q. (꼬리) 독립적 커밋이면 순서는 어떻게 보장하나?
> payment-service가 오케스트레이터로서 직접 순서를 제어한다. STEP 1(포인트 차감) 성공 → STEP 2(카드 결제) 호출 → STEP 3(결제 기록 저장) 순서로, 이전 단계가 성공해야 다음 단계로 넘어간다. 비동기 이벤트가 아니라 동기 호출 체인이라 순서가 코드 흐름 자체로 보장된다.

Q. (꼬리) 그러면 Orchestration SAGA는 동기인가?
> 현재 구현은 동기 Orchestration이다. payment-service가 REST로 point-service를 호출하고 응답을 기다린 뒤 다음 단계로 넘어간다. 그래서 중간에 실패하면 catch에서 바로 보상을 실행할 수 있다. 다만 Orchestration이 반드시 동기인 건 아니다. 대규모 시스템에서는 오케스트레이터가 Kafka로 각 단계에 명령을 보내고 응답 이벤트를 받는 비동기 Orchestration도 있다. 현재 규모에서는 동기가 구현이 단순하고 즉시 응답이 가능해서 적합하다.

Q. 왜 300명, 22,000건인가?
> 인기 공구 오픈 시 동시 접속자를 가정했다. 50명 → 100명 → 200명 → 300명으로 단계적으로 올리면서 어느 시점에 문제가 생기는지 확인했다. 22,000건은 300명이 각각 여러 번 결제하면서 누적된 총 건수이다. 1,000명까지도 시도했지만 로컬 Docker 환경에서 서비스가 OOM으로 죽어서, 안정적으로 측정 가능한 300명을 기준으로 잡았다.

Q. (꼬리) 더 올리려면 어떻게 해야 하나?
> 로컬 Docker 환경의 물리적 한계(CPU, 메모리)가 병목이다. 실제로 1,000명까지 시도했지만 서비스가 OOM으로 죽었다. 더 올리려면 클라우드 멀티노드 환경이 필요하고, 그때 K8s를 다시 도입하면 HPA(자동 확장)와 self-healing이 의미 있어진다.

Q. Choreography가 아니라 Orchestration을 쓴 이유?
> 세 가지 이유이다.
>
> 첫째, 즉시 응답이 필요하다. 사용자가 결제 버튼을 누르면 바로 "성공/실패"를 알려줘야 한다. Choreography는 이벤트를 발행하고 각 서비스가 알아서 처리하는 구조라, 사용자에게 언제 결과를 돌려줄지 보장이 안 된다.
>
> 둘째, 조건 분기가 있다. 잔액 부족이면 결제를 즉시 중단해야 하고, 카드 한도 초과면 카드 단계에서 멈춰야 한다. Choreography에서는 이미 발행된 이벤트를 취소하는 "보상 이벤트"를 또 발행해야 해서 흐름이 복잡해진다. Orchestration은 if문 하나로 끝난다.
>
> 셋째, 보상 흐름이 명확하다. 실패 시 "카드 환불 → 포인트 복구 → 재고 복구 → 적립 포인트 회수"를 역순으로 실행해야 하는데, Orchestration은 catch 블록 안에서 순서대로 호출하면 된다. Choreography에서는 각 서비스가 보상 이벤트를 듣고 각자 판단해야 해서, 보상 누락이나 순서 꼬임이 생길 수 있다. 특히 적립 포인트 회수는 Kafka로 비동기 적립된 포인트를 되돌려야 하는데, 적립이 아직 안 됐을 수 있어서 CompensationOutbox로 재시도가 필요하다.

Q. (꼬리) Kafka에서 같은 파티션이면 순서 보장되지 않나?
> 맞다. 같은 파티션 안에서는 순서가 보장된다. 하지만 문제는 순서가 아니라 "즉시 응답"이다. 사용자가 결제 버튼을 누르면 바로 "성공/실패"를 알려줘야 하는데, Kafka에 이벤트를 넣고 consumer가 처리할 때까지 기다리면 사용자는 결과를 모른 채 대기하게 된다. 결제처럼 동기 응답이 필수인 흐름에는 Orchestration(직접 호출 + 즉시 응답)이 맞다.

Q. 동시성 문제는 어떻게 해결했나?
> 처음에는 MongoDB `findAndModify`로 원자적 차감을 구현했다. 차감 자체의 동시성(마이너스 잔액)은 해결됐지만, 차감과 이력 저장이 별도 연산이라 불일치 가능성이 남았다.

Q. (꼬리) 왜 별도 연산인가? MongoDB 구조 때문인가?
> 그렇다. MongoDB에서 `findAndModify`는 단일 document에 대한 원자적 연산이다. 포인트 차감은 point document, 이력은 point_history document(다른 컬렉션)이라 두 연산을 하나의 원자적 명령으로 묶을 수 없다. multi-document 트랜잭션을 쓰면 가능하지만 Replica Set이 필수이고, MongoDB 공식 문서에서도 "가능하면 트랜잭션 없이 해결하라"고 권장한다. 근본적으로 MongoDB는 매 연산마다 트랜잭션이 필수인 금전 도메인의 설계 철학과 맞지 않아서 MySQL로 전환했다.

Q. (꼬리) MySQL에서는 왜 가능한가?
> MySQL의 `@Transactional`은 여러 테이블 연산을 하나의 트랜잭션으로 묶는다. point 테이블 UPDATE + point_history 테이블 INSERT가 같은 트랜잭션 안에서 실행되고, 하나라도 실패하면 전부 롤백된다. `SELECT FOR UPDATE`로 행에 배타 락을 걸어서 동시 요청이 순차 처리되게 했다. MySQL은 태생부터 트랜잭션이 핵심 기능이라 이 구조가 자연스럽다.

Q. (꼬리) SAGA 보상과 Kafka 적립이 동시에 일어나면 순서가 꼬이지 않나?
> 꼬일 수 있다. 결제 실패 시 SAGA 보상(동기)이 적립 포인트를 회수하려 하는데, Kafka 적립(비동기)이 아직 안 들어왔으면 잔액 부족으로 회수가 실패한다. 하지만 마이너스로 빠지지는 않는다. `SELECT FOR UPDATE`가 잔액 부족이면 차감 자체를 거부하기 때문이다. 회수 실패 건은 CompensationOutbox에 저장되고, 30초마다 재시도한다. Kafka 적립이 들어온 후 재시도하면 회수 성공. 순서를 강제하지 않는 이유는, 동기 보상 흐름에서 비동기 이벤트를 기다리는 건 SAGA의 즉시 응답 원칙에 어긋나기 때문이다. 대신 최종 일관성(Eventual Consistency)으로 어떤 순서든 최종 잔액이 동일하게 수렴하도록 설계했다.

Q. SAGA 적용 후 18건 남았는데, 동시성 도입해도 여전히 남은 건 동시성 문제가 아닌 거 아닌가?
> 맞다. 18건은 동시성 문제가 아니라, 동시 300명 고부하에서 SAGA 보상 REST 호출 자체가 타임아웃 난 케이스다. 포인트는 차감됐고, 보상 호출을 보냈는데 point-service가 응답을 못 한 것이다. 이건 네트워크/서비스 부하 문제이지 동시성 문제가 아니다. 해결하려면 Outbox Pattern(보상 내역을 로컬 DB에 저장 → 스케줄러 재시도)이 필요하다.

## DB 선택

Q. 왜 MySQL로 바꿨나?
> MongoDB는 대규모 비정형 데이터, 유연한 스키마, 수평 확장이 필요한 곳에서 강점을 가지는 DB이다. 상품 카탈로그처럼 카테고리별 스키마가 다른 데이터에는 여전히 최적이다. 하지만 금전 도메인은 매 연산마다 트랜잭션이 필수이고 `SELECT FOR UPDATE` 같은 비관적 락이 필요한데, MongoDB에는 이 기능이 없다. MongoDB의 만들어진 목적과 금전 도메인의 요구사항이 맞지 않았다.

Q. MongoDB 트랜잭션 쓰면 안 되나?
> 쓸 수 있다. 4.0부터 지원한다. 하지만 Replica Set이 필수이고, MongoDB 공식 문서에서도 "트랜잭션은 성능 비용이 크니 스키마 설계로 해결하라"고 권장한다. 쓸 수 있지만 MongoDB의 설계 철학에 반하는 사용법이다. 금전처럼 매 요청마다 트랜잭션이 필요한 도메인에서는 트랜잭션이 태생부터 핵심인 RDBMS가 자연스럽다.

## Kubernetes 도입과 제거

Q. K8s를 도입했다가 뺐다는데?
> 대용량 트래픽 대응을 위해 K8s를 도입했다. 컨테이너를 늘리고 자동 라우팅만 되면 트래픽을 감당할 수 있을 거라고 생각했다. 실제로 단일 노드에서 Pod 스케일 아웃, 트래픽 분배까지 확인했다. 하지만 K8s의 핵심 가치인 self-healing, HPA, 롤링 업데이트는 멀티노드 프로덕션 환경에서 의미 있는 기능이고, 단일 노드에서는 Docker Compose와 실질적 차이가 없었다. 피상적으로 도입하기보다 확실한 필요가 있을 때 쓰는 게 맞다고 판단해서 제거했다.

Q. (꼬리) 그 과정에서 배운 건?
> 세 가지가 있다. 첫째, Eureka + API Gateway가 Docker Compose 서비스명 호출로도 대체 가능하다는 걸 파악하고 제거했다. FeignClient에 `url` 파라미터를 직접 지정하는 방식으로 전환했고, 이건 K8s 제거 후에도 유지된다. 둘째, 같은 RestTemplate 코드가 인프라(Eureka vs K8s)에 따라 다르게 동작한다는 걸 체감했다. Eureka는 요청 단위 분배, kube-proxy는 TCP 연결 단위 분배라서 HTTP Keep-Alive가 변수로 작용했다. 셋째, 기술을 도입할 때 "남들이 쓰니까"가 아니라 "현재 환경에서 이게 해결하는 문제가 있는가"를 먼저 따져야 한다는 것.

Q. Eureka를 왜 제거했나?
> Docker Compose 환경에서 Eureka의 3가지 기능(서비스 디스커버리, 로드밸런싱, 헬스체크)이 Docker Compose 서비스명 호출과 중복이었다. Eureka를 유지하면 서비스마다 JVM에 ~30MB Eureka Client + 별도 Eureka Server(~512MB)가 필요한데, Docker Compose에서는 서비스명으로 바로 호출할 수 있어서 불필요했다.

## 검색

Q. 왜 Elasticsearch를 썼나?
> 세 가지 이유가 있다. 첫째, 한국어 형태소 분석이다. MongoDB `$text`는 띄어쓰기 기반 토큰화만 가능한데, Nori 분석기는 "공동구매"를 "공동"+"구매"로 분리해서 다양한 검색어에 매칭된다. 둘째, 역인덱스(Inverted Index) 자료구조이다. 일반 DB는 document → 단어 방향으로 조회하지만, ES는 단어 → document 방향이라 검색이 O(1)에 가깝다. 셋째, 향후 확장성이다. 자동완성, 오타 교정(fuzzy search), 동의어 처리, 가중치 기반 정렬 등 검색 고도화가 ES 설정만으로 가능하다. MongoDB 텍스트 검색으로는 이런 확장이 어렵다.

Q. 실시간 랭킹은 어떻게 동작하나?
> 검색 시 search-service가 직접 재계산하고, 주문 시에는 Kafka order-events를 통해 수신한 뒤 재계산한다. 점수는 검색횟수×0.4 + 주문량×0.6(최근 1시간 기준). 재계산된 랭킹은 Redis에 캐시하고, WebSocket으로 push한다.

Q. (꼬리) 캐시하는 이유?
> 랭킹 계산은 MongoDB에서 최근 1시간 검색/주문 로그를 집계하는 연산이다. 매 요청마다 이걸 하면 DB 부하가 크다. Redis에 캐시해두면 이벤트 발생 시 갱신하고, REST API로 직접 조회하는 클라이언트(WebSocket 미연결 상태)에게는 캐시된 결과를 반환해서 DB 부하를 줄인다.

Q. 자동완성에서 "로그 없이 ES 조회만"이 무슨 뜻인가?
> 자동완성(`/suggest`)은 사용자가 타이핑하는 중간 입력("과", "과자", "과잠")에 대해 ES에서 후보를 조회해서 보여주기만 한다. 이 중간 입력을 검색 로그(MongoDB search_log 컬렉션)에 저장하면 랭킹이 오타와 미완성 키워드로 오염된다. 실제 검색(`/search`)에서만 로그를 저장하고 랭킹에 반영한다.

Q. WebSocket의 단점은?
> 서버가 모든 접속자와 연결을 유지해야 하므로 메모리를 먹는다. 접속자 10만 명이면 10만 개 연결을 들고 있어야 한다. 또한 연결이 끊기면 재연결 로직이 필요하고, 로드밸런서 뒤에서 sticky session 문제도 생길 수 있다. 그래서 60초 폴링을 폴백으로 두어, WebSocket이 끊겨도 랭킹이 멈추지 않도록 했다.

## 보안

Q. HMAC을 왜 떠올렸나?
> 결제 시스템을 구현하면서 "프론트에서 올라온 금액을 서버에서 재계산한다"는 건 알았는데, 그래도 서버 간 통신에서 데이터가 위변조되면 어떡하나 생각이 들었다. PG사 연동 자료를 조사하다 보니 토스페이먼츠가 웹훅에 HMAC-SHA256 서명을 쓰고 있었고, "외부에서 온 요청을 신뢰할 수 있는가"라는 문제를 HMAC으로 해결하는 걸 보고 적용하게 됐다.

Q. 서명값의 자료구조? 결제 기록 컬럼 구성?
> HMAC-SHA256의 결과는 64자 hex 문자열이다. payment 테이블의 `hmac_signature` 컬럼(VARCHAR)에 저장한다. 결제 기록 전체 컬럼은: id, order_id, user_id, title, quantity, unit_price, total_amount, point_used, card_amount, status, payment_method, card_id, approval_number, hmac_signature, created_at, completed_at이다.

Q. 서명값은 따로 보호 안 하나?
> 서명값 자체는 Secret Key 없이는 위조할 수 없으므로 평문 저장해도 괜찮다. 중요한 건 Secret Key이다. 현재는 application.yml에 평문으로 있지만, 실제 운영에서는 환경변수나 HashiCorp Vault로 관리해야 한다.

## 기술 깊이

Q. @Transactional이 뭔가?
> DB의 트랜잭션(begin/commit/rollback)을 자동으로 처리해주는 Spring 어노테이션이다. 메서드 안의 모든 SQL이 하나의 트랜잭션으로 묶여서, 하나라도 실패하면 전부 롤백된다. 단, 하나의 DB 커넥션 안에서만 동작하는 로컬 트랜잭션이라, 다른 서비스의 다른 DB는 묶을 수 없다. 그래서 MSA에서는 SAGA가 필요하다.

Q. SELECT FOR UPDATE 경험과 한계?
> 500P 잔액에 10건 동시 요청을 보냈을 때, `SELECT FOR UPDATE` 없이는 10건 전부 성공하고 잔액이 -500P가 됐다. 적용 후에는 정확히 5건만 성공하고 나머지 5건은 "잔액 부족"으로 거절됐다. 한계는 행 락 대기 시간이다. 동시 요청이 많으면 락을 잡으려고 대기하는 시간이 늘어나 응답 속도가 느려진다. 실제로 SAGA 부하 테스트에서 `findAndModify`(MongoDB)보다 `SELECT FOR UPDATE`(MySQL)가 document-level lock vs row-level lock 경합으로 응답 시간이 약간 늘어났다. 처리량(throughput)이 중요한 경우 낙관적 락(@Version + 재시도)을 검토할 수 있다.

Q. Kafka를 왜 썼나?
> 1건 주문 → 3개 서비스(payment, point, search)에 동시 전달하는 팬아웃 구조이다. Kafka는 토픽 기반 pub/sub이라 consumer를 추가해도 producer 코드를 안 바꿔도 된다. 또한 메시지가 디스크에 저장되어 consumer가 죽었다 살아나도 못 읽은 메시지를 이어서 처리(리플레이)할 수 있다. RabbitMQ도 팬아웃은 가능하지만 exchange 설정이 필요하고, 메시지가 기본적으로 consumer가 읽으면 사라져서 리플레이가 안 된다.

Q. WebSocket을 왜 썼나?
> 두 가지 이유이다. 첫째, 폴링은 클라이언트가 주기적으로 요청을 보내야 해서 접속자가 많으면 불필요한 트래픽이 폭주한다. 둘째, 폴링은 주기(예: 5초)마다 조회하니까 랭킹 변동이 최대 5초 늦게 반영되는데, WebSocket은 이벤트 발생 즉시 push하니까 실시간 정확도가 높다.
