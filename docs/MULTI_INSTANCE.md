# Kubernetes 도입과 제거 — 왜 넣었고, 왜 뺐는가

---

## 처음 생각: "컨테이너를 늘리면 되지"

대용량 트래픽을 대응하고 싶었다. Docker Compose만 써본 입장에서, 해결책은 단순해 보였다. 컨테이너(인스턴스)를 여러 개 띄우고, 앞에서 자동으로 라우팅만 해주면 된다고 생각했다. API Gateway의 라운드로빈이든, K8s의 kube-proxy든, 요청을 분배해주는 무언가만 있으면 트래픽이 늘어도 대응할 수 있을 거라고.

물이 넘치면 그릇을 하나 더 가져오면 되지 — 정도의 발상이었다.

---

## 도입 시도: K8s 전환

그래서 Docker Compose → Kubernetes로 전환했다. 실제로 K8s 환경에서 point-service를 3대로 스케일 아웃하고, 트래픽이 분배되는 것까지 확인했다. 그 과정에서 얻은 것도 있었다.

- Eureka + API Gateway가 K8s Service DNS + kube-proxy와 완전히 중복된다는 걸 파악하고 제거했다
- Java RestTemplate의 HTTP Keep-Alive가 kube-proxy의 TCP 연결 단위 분배와 충돌해서 한 Pod에 트래픽이 고착되는 문제를 발견하고, `Connection: close` 헤더로 해결했다
- Docker Compose에서 잘 되던 Kafka가 K8s에서 CrashLoopBackOff를 일으킨 원인(K8s Service 환경변수 자동 주입 충돌)을 찾아서 해결했다

기술적으로 배운 건 있었다. 하지만 문제는 그 다음이었다.

---

## 현실: 물리적 한계

막상 "대용량 트래픽 대응"을 하려고 보니, 컨테이너를 늘리는 것만으로는 안 됐다. K8s에 대해 어렴풋이만 알고 있었기 때문에, **스케일 아웃이 물리적인 노드(서버)가 더 필요하다는 걸 인지하지 못했다.** 단일 노드(맥북 1대)에서 Pod을 아무리 늘려봤자 CPU와 메모리는 같은 머신을 나눠 쓰는 것이고, 진짜 트래픽 분산이 아니라 같은 그릇 안에서 칸막이를 세운 것에 불과했다.

멀티노드 클러스터를 구성하려면 물리 서버가 최소 2~3대 필요한데, 집에 남는 노트북이 없었다. 클라우드를 쓰면 되지만, 그건 이 프로젝트의 범위를 넘어서는 비용과 인프라 운영 영역이었다.

---

## 제거 결정

K8s 구동 자체는 됐다. 단일 노드에서 Pod 띄우고, Service 만들고, 분배 확인까지 했다. "그냥 썼다"는 적용할 수 있었다.

하지만 다른 사람들이 K8s를 공부하고 적용한 사례를 읽다 보니, 내가 도입한 건 피상적인 선택에 불과했다. 온프레미스 멀티노드 클러스터, ArgoCD GitOps, Prometheus 모니터링 — 이런 걸 보면서, 내 K8s 적용은 Docker Compose에서 할 수 있는 걸 K8s 문법으로 옮겨 적은 것에 가깝다는 걸 인정하게 됐다.

**얕은 이유로 도입하기보다는 확실한 필요에 의한 선택을 하자**는 가치관이 있다. K8s를 쓸 명분이 — self-healing, HPA, 롤링 업데이트 — 전부 멀티노드 + 프로덕션 환경에서 의미 있는 기능인데, 단일 노드 로컬 환경에서는 그 필요가 성립하지 않았다. 결국 제거했다.

---

## 남은 것

K8s는 제거했지만, 그 과정에서 얻은 건 남아 있다.

- **Eureka 제거 판단**: K8s가 아니더라도, Docker Compose에서 서비스명으로 호출할 수 있는 환경에서 Eureka는 불필요하다는 걸 알게 됐다. FeignClient에 `url` 파라미터를 직접 지정하는 방식으로 전환했고, 이 구조는 K8s 제거 후에도 그대로 유지된다.
- **인프라가 바뀌면 코드 동작이 달라진다는 감각**: 같은 RestTemplate 코드가 Eureka 환경에서는 요청 단위로 분배되고, K8s 환경에서는 TCP 연결 단위로 분배된다는 차이. 이건 직접 부딪혀봐야 알 수 있는 것이었다.
- **"왜 필요한가"를 먼저 묻는 습관**: 기술을 도입할 때 "남들이 쓰니까"가 아니라 "현재 환경에서 이게 해결하는 문제가 있는가"를 먼저 따져야 한다는 것.

현재는 Docker Compose 기반으로 6개 서비스 + 인프라 전체를 배포하고 있다. 대용량 트래픽 대응이 필요해지는 시점이 오면, 그때 멀티노드 환경과 함께 K8s를 다시 도입할 것이다.
